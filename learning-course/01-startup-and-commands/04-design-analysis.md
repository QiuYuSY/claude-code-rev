# 设计分析：为什么这样设计

## 一、为什么入口要分层，而不是一个 `main()`

### 问题

如果所有逻辑都写在一个 `main()` 里，会怎样？

### 回答

短期看起来简单，长期会产生 4 个问题：

1. 所有调用路径共享同样的初始化成本
2. 特殊模式越来越难插入
3. 启动性能优化没有着力点
4. 入口会同时承担环境修正、模式判断、业务初始化和命令执行

Claude Code 的分层设计，本质上是在给复杂度分舱。

## 二、为什么 `cli.tsx` 要像网关一样工作

### 问题

为什么不直接在 `main.tsx` 内部 if/else 判断各种特殊参数？

### 回答

因为很多路径根本不应该进入 `main.tsx`。

例如：

- `--version`
- daemon worker
- remote-control bridge
- template jobs
- environment runner

它们的共同点是：

- 目标明确
- 所需依赖少
- 不需要完整 REPL

把这些路径留在 `cli.tsx`，可以让 `main.tsx` 更专注于完整运行时。

## 三、为什么 `main.tsx` 里会出现很多“时机”相关逻辑

### 问题

为什么这里有这么多“先做 A 再做 B，但 C 可以并行”的逻辑？

### 回答

因为大型 CLI 启动优化本质上就是时机管理。

例如：

- 什么时候设置环境变量，才能让后续 import 读到正确值
- 什么时候注册 bundled skills，才能让 `getCommands()` 拿到正确缓存
- 什么时候执行 `init()`，才能避免 help 模式做无意义初始化

这类逻辑看起来琐碎，但它们正是成熟系统和 demo 项目的分界线。

## 四、为什么 `commands.ts` 必须是注册中心

### 问题

为什么不维持一个简单数组，每次改命令就手动补进去？

### 回答

因为这个系统的命令来源已经天然多元：

- 内建
- 插件
- Skills
- Workflows
- 动态技能

来源一多，就需要一个统一出口来处理：

- 加载
- 排序
- 过滤
- 可用性判断
- 模型视角与用户视角的区分

注册中心的意义就在这里。

## 五、这个设计最强的地方

### 1. 把性能意识写进了结构里

不是“优化时才想起性能”，而是从入口层就开始做性能结构设计。

### 2. 把复杂度拆成层，而不是堆在函数里

层级关系比单纯的代码长度更重要。

### 3. 把扩展性考虑得足够早

`commands.ts` 的设计明显是按“长期会有更多命令来源”来构建的。

## 六、这个设计的维护风险

### 风险 1：`main.tsx` 继续膨胀

如果所有初始化判断继续都往这里堆，它会越来越像“只有作者自己懂的启动神殿”。

### 风险 2：fast-path 太多，入口难以统一测试

每新增一个 fast-path，都会增加一条新的启动支路。

### 风险 3：命令不可见问题排查链过长

一个命令之所以没出现，可能是：

- 没被加载
- 加载了但不可用
- 被 feature gate 关了
- 被 auth 过滤了
- 被 `isEnabled()` 过滤了
- 被技能视图规则过滤了

这就是平台化带来的复杂度代价。

## 七、可迁移经验

### 什么时候应该模仿这套设计

适合：

- 复杂 CLI
- 多运行模式
- 有插件或扩展系统
- 有交互与 headless 双路径

### 什么时候不应该模仿

不适合：

- 单一功能 CLI
- 没有长期演化计划的小工具
- 没有插件、没有多模式、没有重初始化的小项目

## 八、这一课真正想让你学会什么

不是“记住这几个函数”，而是学会这个判断：

当一个 CLI 开始变复杂时，最先要做的不是继续堆命令，而是把启动链路和注册中心做出来。
