# 第五课概览

## 核心问题

这节课围绕 4 个问题展开：

1. Claude Code 的扩展能力为什么至少要分成 MCP、插件、Skills 三条线？
2. 为什么外部能力接入不能只靠手写内建命令？
3. 为什么插件命令要统一转成 `Command`？
4. 为什么最终还是要回到 `commands.ts` 的统一汇总？

## 你会读到哪些文件

- `src/services/mcp/client.ts`
- `src/services/mcp/config.*`
- `src/utils/plugins/loadPluginCommands.ts`
- `src/commands.ts`

## 先给结论

Claude Code 的扩展架构不是“顺手加了插件系统”，而是已经形成了平台化形态：

- MCP：协议型外部能力接入
- 插件：本地内容与命令扩展
- Skills：面向模型的结构化能力入口

这三条线共同说明：
系统已经把“扩展”视为一等公民。

## 这一课最值得学的地方

### 1. 学协议接入层

MCP 客户端展示了一个成熟系统如何接入外部能力，同时处理认证、资源、超时和会话失效。

### 2. 学“内容即扩展”

插件系统展示了 markdown/frontmatter 如何直接生成命令，这比“只能写 JS 插件”更开放。

### 3. 学统一注册中心

扩展来源再多，最终仍然要统一汇总、排序、过滤和暴露。

## 推荐你带着什么视角读

读这节课时，反复问自己：

- 为什么 MCP 不只产出 tools，还产出 resources 和 skills？
- 为什么插件命令不是直接执行，而要先转成统一 `Command`？
- 为什么 `getCommands()` 才是平台化真正收口的地方？

## 学完后的判断标准

如果你能完整说出这句话，方向就是对的：

“Claude Code 通过 MCP 接协议能力，通过插件接本地扩展，通过 Skills 接模型能力入口，并最终用统一注册中心把这些来源收束成同一套命令与能力视图。”
