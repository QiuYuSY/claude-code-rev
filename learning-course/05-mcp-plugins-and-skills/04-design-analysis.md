# 设计分析：为什么这样设计

## 一、为什么 MCP 不能只是“多几个内建命令”

### 问题

为什么 Claude Code 要专门有一层 MCP client，而不是把外部服务全部包装成内建工具？

### 回答

因为协议接入层解决的问题远不止命令调用，还包括：

- transport
- auth
- timeout
- resource discovery
- session 失效恢复
- 结果持久化

这些都是平台级问题，不适合散落在各个命令里。

## 二、为什么插件命令选择 markdown/frontmatter 方案

### 问题

为什么插件不只接受 JS 代码，而要支持 markdown/frontmatter？

### 回答

因为很多扩展本质上是：

- prompt 内容
- 元信息
- 调用约束

对这类扩展来说，内容驱动比代码驱动更低门槛，也更容易审阅和维护。

## 三、为什么一切最终都要转成 `Command`

### 问题

为什么插件命令、skills、workflow 最终都要进入统一 `Command` 抽象？

### 回答

因为平台扩展最重要的不是来源一致，而是出口一致。

只要出口统一，系统就能统一处理：

- 过滤
- 排序
- 展示
- 可用性判断
- 模型可调用性判断

## 四、为什么统一注册中心是平台化关键

### 问题

多来源扩展为什么不能各自维护自己的命令列表？

### 回答

因为那样会很快出现：

- 重复命令
- 不一致的排序
- 不一致的可见性规则
- 用户和模型看到的能力集合脱节

统一注册中心就是平台的收口点。

## 五、这个设计最强的地方

### 1. 把扩展当成系统主线，而不是附属能力

这让 Claude Code 具备长期演化成生态平台的可能。

### 2. 协议接入层和内容扩展层分工明确

MCP 解决外部系统接入，插件解决本地扩展，这种边界很清晰。

### 3. 模型视图和用户视图被区分开来

这让 “命令” 和 “技能” 可以有不同的暴露标准。

## 六、这个设计的维护风险

### 风险 1：扩展来源越多，调试链路越长

一个命令没出现，可能卡在：

- 来源加载
- 元信息质量
- availability
- `isEnabled()`
- 技能视图过滤

### 风险 2：MCP 协议接入复杂度高

外部系统的不稳定、认证过期、资源差异都会增加平台复杂度。

### 风险 3：元数据质量成为系统质量的一部分

插件和技能越依赖内容定义，描述、参数、约束信息的质量就越关键。

## 七、可迁移经验

### 什么时候适合模仿这套设计

适合：

- 你在做平台型 AI 工具
- 需要接多个外部系统
- 希望允许用户或团队自定义扩展
- 需要把能力同时暴露给用户和模型

### 什么时候不适合

不适合：

- 单一业务工具
- 不准备开放扩展
- 需求边界很窄、生命周期很短

## 八、这节课真正想让你学会什么

不是记住 MCP 或插件的 API，而是学会这个判断：

当 AI 产品进入平台化阶段，真正难的不是“支持扩展”，而是“让不同来源的扩展能力在统一抽象和统一治理下稳定共存”。
