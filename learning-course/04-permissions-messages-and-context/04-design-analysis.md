# 设计分析：为什么这样设计

## 一、为什么权限不能只是一个确认弹窗

### 问题

为什么权限系统要深入工具池、模式切换和运行时上下文？

### 回答

因为权限影响的不只是“是否放行”，还影响：

- 模型能看到什么能力
- 自动模式是否安全
- 背景任务是否能继续
- 子代理能继承什么能力

如果权限太晚才生效，模型的推理路径都会被污染。

## 二、为什么消息不能只是字符串数组

### 问题

为什么要有那么多 message 工厂和 message 类型？

### 回答

因为系统里发生的不只是“说话”，还包括：

- 工具调用
- 工具结果
- 系统事件
- 恢复边界
- 摘要和元消息

字符串数组表达不了这些语义。

## 三、为什么内部历史和 API 输入必须分层

### 问题

为什么还要有 `normalizeMessagesForAPI(...)` 这一层？

### 回答

因为内部运行时为了交互、恢复和调试，会保存更多信息；而 API 输入必须：

- 语义稳定
- 结构合法
- 剔除 display-only 内容

这两者天然不一样。

## 四、为什么上下文一定要显式装配

### 问题

为什么不在调用模型前临时把各种信息拼成一个 prompt？

### 回答

因为上下文来源：

- 多
- 开销不同
- 缓存策略不同
- 生命周期不同

不显式装配，就会出现重复注入、顺序混乱和难以测试的问题。

## 五、这个设计最强的地方

### 1. 权限被前置成运行时边界

这让系统的“能见度”和“可执行性”更一致。

### 2. 消息成为统一历史语言

这让 UI、工具、恢复、模型输入都能围绕同一条主线运转。

### 3. 上下文装配可缓存、可诊断、可独立演化

这是长期 agent 系统非常重要的能力。

## 六、这个设计的维护风险

### 风险 1：消息模型继续膨胀

一旦新类型持续加入，message 工厂和 normalize 规则会越来越复杂。

### 风险 2：权限规则复杂度持续增长

尤其当 auto mode、dangerous permissions、base tools、disk rules 继续扩展时，会增加理解成本。

### 风险 3：上下文来源继续增多

如果没有明确分层，context assembly 很容易失控。

## 七、可迁移经验

### 什么时候适合模仿这套设计

适合：

- 多工具 agent
- 有自动模式或危险操作
- 有长期会话和恢复需求
- 需要把系统信息持续注入模型

### 什么时候不适合

不适合：

- 单轮 prompt 调用
- 没有工具和权限需求
- 没有长期上下文积累

## 八、这节课真正想让你学会什么

不是记住哪些函数，而是学会这个判断：

当 agent 系统开始同时追求能力、可控性和长期会话时，权限、消息和上下文必须被一起设计，而不能各自为政。
