# 设计分析：为什么这样设计

## 一、为什么 `launchRepl()` 要这么薄

### 问题

为什么不把 REPL 初始化逻辑都塞进 `launchRepl()`？

### 回答

因为入口连接层最怕职责扩张。

一旦它同时负责：

- 加载组件
- 初始化状态
- setup 流程
- 错误处理

它就会变成新的巨石入口。

保持 `launchRepl()` 薄，能让职责更稳定。

## 二、为什么 `App.tsx` 更像容器而不是业务组件

### 问题

为什么 `App.tsx` 主要是在包 Provider，而不是写业务 UI？

### 回答

因为顶层组件最重要的职责不是展示，而是提供：

- 状态根
- 统计与性能上下文
- bootstrap 错误边界

这是一种典型的“根容器”和“业务组件”分离设计。

## 三、为什么 `interactiveHelpers.tsx` 这么重要

### 问题

为什么要专门有一层 helper 来处理 render、setup、exit、shutdown？

### 回答

因为交互生命周期本身就是一等复杂度。

如果把这些逻辑分散在各个命令、各个组件和各个入口里，系统会很快失去一致性。

## 四、为什么 `AppState` 会这么大

### 问题

`AppState` 这么大，是不是设计有问题？

### 回答

不一定。

这里的大，很大程度上是在反映真实系统复杂度：

- 长会话
- 多任务
- 多 agent
- MCP
- 插件
- remote / bridge
- 通知与 elicitation

如果这些能力都存在，那么状态中心不可能很小。

真正要担心的不是“大”，而是：

- 是否有清晰状态域
- 是否有明确消费边界
- 是否有清晰生命周期

## 五、这个设计最强的地方

### 1. UI 成为了运行时的一部分

这让 Claude Code 能自然支持：

- setup 流程
- trust 边界
- 权限交互
- 多任务与 agent 视图
- remote / bridge 状态

### 2. 状态总表暴露了产品真实能力

这让维护者和学习者都能更快识别系统核心。

### 3. 顶层分层还算克制

`launchRepl()`、`App.tsx`、`interactiveHelpers.tsx` 各自职责都比较明确。

## 六、这个设计的维护风险

### 风险 1：`AppState` 继续膨胀

如果新增能力没有状态域 discipline，状态中心会越来越难懂。

### 风险 2：交互编排逻辑继续外溢

如果未来 setup / remote / bridge / onboarding 继续增长，`interactiveHelpers.tsx` 也可能膨胀。

### 风险 3：UI 和运行时耦合很深

这是收益，也是代价。好处是体验统一，坏处是重构任一侧都会影响另一侧。

## 七、可迁移经验

### 什么时候适合模仿这套设计

适合：

- 终端产品有长期会话
- 有权限与 setup 流程
- 有插件或远程能力
- 有多任务或 agent 协作

### 什么时候不适合

不适合：

- 纯单轮命令工具
- 没有任务和长期状态
- 不需要复杂 setup/trust 流程

## 八、这节课真正想让你学会什么

不是学 Ink 语法，而是学会这个判断：

当终端产品开始承担长期交互和运行时编排，它就不再是“命令输出器”，而需要前端式状态管理和交互 orchestration。
