# 逐段精读：`README.md -> shims/ -> 课程主干图解`

这份文档按“先识别仓库性质，再看兼容层，再回到主干图解”的顺序带你读第六课的关键材料。

---

## 一、根目录 `README.md`

### 第一眼应该看什么

先不要看运行方式，而要看仓库性质说明。

README 里非常明确地说了几件事：

- 这是 restored source tree
- 它不是 original upstream state
- 有些模块无法恢复，只能用 compatibility shims 或 degraded implementations

### 为什么这几句最重要

因为它们定义了你阅读这个仓库时最重要的前提：

你面对的不是“原作者精心整理后的源码”，而是“尽可能恢复出来且能继续工作的源码树”。

### 还要注意什么

README 同时也告诉你：

- 真实 CLI bootstrap 已经恢复
- 部分 bundled skills 已恢复
- Chrome MCP / Computer Use MCP 的兼容层已补到“可用”

### 这说明什么

说明主干并不是一片废墟，而是已经有相当多值得学习的结构恢复出来了。

---

## 二、`shims/` 目录

### 为什么这一层要专门看

因为它最容易让读者误判：

- 看到 shim 很多，就以为原系统本来就是这么实现的
- 看到 degraded，就误以为整个系统设计都很“临时”

### 你应该怎么理解 `shims/`

更准确的理解是：

- 这里是恢复工程层
- 目标是补洞与兼容
- 不是产品主干的最佳表达

### 这里最适合学什么

不是学产品架构，而是学一种工程现实：

复杂系统在源码不完整时，往往要靠 shim 才能重新建立可运行性。

### 什么时候值得深入看 shim

适合在两种情况下看：

1. 你要研究某个具体能力为什么被降级
2. 你要做恢复工程或兼容层设计参考

如果你的目标是学习 Claude Code 主架构，就不该优先钻 shim。

## 三、回到课程图解

这里是第六课最重要的动作：

当你已经确认：

- 仓库是恢复版
- 兼容层存在

下一步不是去追更多兼容代码，而是回到三张主干图：

- 启动链路图
- 运行时循环图
- 扩展机制图

### 为什么一定要回到图解

因为图解能帮你重新聚焦主干：

- 系统如何启动
- 运行时如何闭环
- 扩展如何收口

### 这一步的真正意义

是把你的注意力从“恢复细节”拉回“产品骨架”。

## 五、把恢复版阅读方法总结成一句话

```text
先确认仓库性质
-> 再区分主干与兼容层
-> 再回到架构主线
-> 最后输出自己的设计判断
```

## 这一套方法回答了什么更大的问题

回答：
它回答的是“面对一个不完美但信息量极高的大仓库，如何不被局部噪音牵着走，而真正学到可迁移的工程能力”。
