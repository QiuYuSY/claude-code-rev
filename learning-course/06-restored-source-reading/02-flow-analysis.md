# 恢复版仓库阅读与复盘流程分析

## 主流程

```text
README.md
  -> 识别仓库性质：restored source tree
      -> 确认主干已恢复、兼容层仍存在
          -> 区分三类内容
              -> 架构主干
              -> 主干支撑模块
              -> 恢复/兼容噪音
                  -> 结合课程图解复盘
                      -> 输出自己的架构图与设计评估
```

## 第 1 层：先确认仓库性质

根目录 `README.md` 已经明确告诉你：

- 这是 restored Claude Code source tree
- 部分模块不可恢复，只能用 shim / degraded implementation
- 项目目标是“可运行、可继续修复”，而不是“精确等同 upstream”

### 为什么这一步关键

因为如果你忘了它是恢复版，就会错误地把很多兼容逻辑当成原始产品设计。

## 第 2 层：区分三类内容

### A 类：架构主干

例如：

- 启动链路
- query loop
- Tool / tools 抽象
- AppState
- MCP / plugin / skills 主线

### B 类：主干支撑模块

例如：

- messages
- permissions
- context
- interactive helpers

### C 类：恢复/兼容噪音

例如：

- `shims/`
- `vendor/`
- 明显带有 fallback / compatibility / degraded 标记的路径

## 第 3 层：先用图解复盘，再追细节

现在课程里已经有三张图：

- 启动链路图
- 运行时循环图
- 扩展机制图

### 为什么要先看图

因为大仓库最怕在局部函数里迷路。图能帮助你判断：

- 这段代码属于哪条主线
- 它是主干还是边角

## 第 4 层：形成设计评价

真正的复盘不是“把文件再念一遍”，而是：

- 总结最值得学的设计
- 总结最危险的复杂度来源
- 判断哪些设计适合迁移，哪些不适合

## 这一整套方法在解决什么问题

### 问题 1：恢复版噪音会污染判断

解决方式：
先确认仓库性质，再分类阅读。

### 问题 2：复杂项目容易没有骨架感

解决方式：
先看主干图解，再回看细节。

### 问题 3：学习最后容易停在“我看过”

解决方式：
强制输出架构图和设计评估。

## 阶段性总结

如果你只记一句话，应该记这句：

“恢复版仓库的关键不是把所有兼容逻辑都看懂，而是先抓主干、再辨噪音、最后产出自己的结构化判断。”
