# 企业内部设计 Agent 时，如何借鉴 Claude Code 的设计

这份文档不是源码学习讲义，而是一份更适合企业内部汇报、方案讨论和设计评审的参考稿。

它要回答的问题是：

“如果我们在企业内部设计自己的 Agent 系统，Claude Code 里有哪些设计值得借鉴？哪些不应该直接照搬？应该如何落地？”

## 一页结论

如果只能讲一句话，可以这样概括：

Claude Code 最值得企业借鉴的不是某个具体功能，而是它把 Agent 系统当成“长期运行的交互式平台”来设计，而不是把它当成“一次模型调用 + 几个工具”的简单封装。

企业内部设计 Agent 时，最值得借鉴的 5 个点是：

1. 分层启动与多模式入口
2. 以 query loop 为核心的运行时主循环
3. 权限前置到工具暴露层，而不是事后确认
4. 用统一状态中心承载长期会话复杂度
5. 用统一注册中心治理多来源扩展能力

## 这份文档适合谁

- 企业内部做 AI Agent 平台设计的人
- 要做方案汇报、架构评审的人
- 想从 Claude Code 中提炼“可迁移工程原则”的人

## 一、为什么 Claude Code 值得企业借鉴

很多 Agent 产品看起来很强，但企业内部真正落地时最容易失败的原因并不是模型不够强，而是：

- 运行时不可控
- 权限边界不清晰
- 工具扩展越来越乱
- 长会话状态难以维护
- 无法兼顾 CLI、服务端、远程、自动化多入口

Claude Code 值得借鉴，是因为它在工程上正面解决了这些问题。

### 它最有价值的地方不是“能写代码”

而是它把一个 coding agent 做成了：

- 有启动分层
- 有明确 runtime loop
- 有权限和消息边界
- 有状态中心
- 有平台化扩展架构

这对企业内部 Agent 平台设计非常有参考意义。

## 二、企业最值得借鉴的 5 个设计原则

## 1. 不要把 Agent 做成“单次请求函数”，要做成“运行时”

### Claude Code 的做法

Claude Code 不是用户发一句话、系统回一句话这么简单，而是通过 `query.ts` 中的主循环来处理：

- 模型流式输出
- 工具调用
- 工具结果回填
- token budget
- compact
- 中断与继续

### 企业可借鉴原则

如果你的 Agent 会出现下面任意三项：

- 多轮会话
- 工具调用
- 流式输出
- 用户中断
- 长任务执行
- 子任务/多 Agent 协作

就应该把它设计成运行时主循环，而不是继续堆 service helper。

### 汇报可用表述

“我们的 Agent 不应只建模为一次推理请求，而应建模为一个带状态转移、工具编排和恢复能力的运行时流程。”

## 2. 权限要前置到能力暴露层，不要只做事后审批

### Claude Code 的做法

Claude Code 在启动和会话初始化阶段就构建 `ToolPermissionContext`，权限不仅影响：

- 工具能否执行

还影响：

- 模型能看到哪些工具
- 自动模式能否开启
- 哪些危险规则必须被剥离

### 企业可借鉴原则

企业内部 Agent 最危险的做法之一，就是让模型先看到所有能力，再在执行时临时拦。

更好的做法是：

- 先根据用户身份、项目、环境、策略确定能力边界
- 再把“当前允许暴露的工具集合”交给模型

### 汇报可用表述

“权限治理不应只是执行时审批，而应进入能力暴露层，确保模型推理路径本身受到治理。”

## 3. 统一消息模型，而不是把聊天、工具、系统事件分开存

### Claude Code 的做法

Claude Code 用统一的 `message` 系统承载：

- 用户输入
- 模型输出
- tool use
- tool result
- system message
- attachment
- compact boundary

### 企业可借鉴原则

企业内部 Agent 系统一旦涉及：

- 工具调用
- 审计
- 恢复
- 回放
- 多渠道接入

就不应该只用“聊天文本”当历史。

更合适的做法是使用统一消息/事件模型。

### 汇报可用表述

“统一消息模型是 Agent 可审计、可恢复、可重放、可治理的基础，不只是 UI 所需的数据结构。”

## 4. 长会话系统一定会走向状态中心

### Claude Code 的做法

Claude Code 通过 `AppState` 把下列复杂度统一收束：

- 会话与模型状态
- 权限状态
- 任务与 agent 状态
- MCP 与插件状态
- remote / bridge 状态
- 通知与 elicitation

### 企业可借鉴原则

如果企业 Agent 产品要支持：

- 长会话
- 后台任务
- 多 Agent
- 多入口客户端
- 插件扩展

那么局部状态已经不够，需要统一状态中心或统一会话状态模型。

### 汇报可用表述

“随着 Agent 从单轮推理走向长期协作，会话状态会成为平台核心资产，必须有统一状态模型。”

## 5. 多来源扩展必须统一治理

### Claude Code 的做法

Claude Code 同时支持：

- 内建命令
- Skills
- 插件命令
- 工作流
- MCP 工具与资源

但最终都要回到统一注册中心 `commands.ts` 处理加载、排序、过滤、可见性和模型可调用性。

### 企业可借鉴原则

企业内部 Agent 一旦进入平台化阶段，最容易失控的地方不是模型，而是扩展生态。

如果没有统一注册中心和统一元数据抽象，很快就会出现：

- 重复能力
- 无法排查
- 不一致的权限
- 不一致的展示

### 汇报可用表述

“平台化 Agent 的核心不是支持扩展，而是统一治理扩展。”

## 三、企业不应该直接照搬的地方

Claude Code 很值得学，但不是所有设计都适合企业原样照搬。

## 1. 不要直接照搬它的全部终端交互形态

Claude Code 的 REPL/TUI 很强，但企业内部很多场景可能更适合：

- Web 控制台
- IDE 面板
- IM/工单式入口
- 自动化触发入口

应该借鉴的是：

- 运行时与状态设计

而不是必须照搬：

- 终端 UI 形态

## 2. 不要一开始就做全量平台化

Claude Code 的能力层很厚：

- 多模式入口
- 多工具
- 多扩展来源
- remote/bridge

企业内部如果第一阶段需求还不明确，直接照搬会导致过度设计。

更合理的方式是分阶段建设。

## 3. 不要把开源/个人工作流工具直接等同于企业治理工具

企业内部要额外考虑：

- 身份与权限体系
- 数据分级
- 审计日志
- 合规与留痕
- 多租户隔离
- 组织级策略

Claude Code 给你的更多是工程结构灵感，而不是现成企业治理模板。

## 四、推荐的企业 Agent 架构借鉴方案

如果用 Claude Code 作为参考，我建议企业内部 Agent 平台按 4 层设计。

## 第 1 层：入口层

可支持：

- Web UI
- IDE 插件
- CLI
- IM 机器人
- 自动化任务触发

借鉴 Claude Code 的地方：

- 分层启动
- 不同模式的入口分流

## 第 2 层：运行时层

负责：

- query loop
- 工具编排
- 中断与恢复
- 输出流式处理
- 预算和上下文控制

借鉴 Claude Code 的地方：

- 显式 runtime loop
- message 主线
- tool execution 闭环

## 第 3 层：治理层

负责：

- 权限上下文
- 工具暴露控制
- 审计与可追踪
- 安全策略
- 数据边界

借鉴 Claude Code 的地方：

- 权限前置
- 规则驱动的能力暴露

## 第 4 层：扩展层

负责：

- 内建工具
- 平台插件
- MCP/协议接入
- 组织自定义技能

借鉴 Claude Code 的地方：

- 统一注册中心
- 多来源扩展统一抽象

## 五、如果要做汇报，建议这样讲

下面是一套可以直接拿去汇报的 6 页结构。

## 第 1 页：问题定义

讲：

- 企业 Agent 不只是模型效果问题
- 真正难点在运行时可控、权限治理、扩展治理、长会话状态

## 第 2 页：为什么参考 Claude Code

讲：

- 它不是简单聊天产品
- 它在工程上完整处理了 Agent 运行时问题

## 第 3 页：我们最该借鉴的 5 个设计

讲：

- 分层启动
- query loop
- 权限前置
- 统一消息模型
- 统一扩展注册中心

## 第 4 页：我们不会照搬的部分

讲：

- 不会直接照搬终端交互形态
- 不会第一阶段就做全量平台化
- 不会忽略企业权限与审计

## 第 5 页：我们的目标架构

讲：

- 入口层
- 运行时层
- 治理层
- 扩展层

## 第 6 页：实施路径

建议分 3 期：

### 第一期：先做单 Agent + 核心工具 + 权限边界

目标：

- 可控
- 可审计
- 可用

### 第二期：补长期会话、任务与恢复

目标：

- 支持更长链路任务
- 支持中断与恢复

### 第三期：补平台化扩展

目标：

- 插件/MCP
- 多入口
- 团队级技能与规范

## 六、面向管理层的结论

如果面对管理层，需要把话讲得更短：

“Claude Code 最值得我们借鉴的，不是它会不会写代码，而是它把 Agent 做成了一个可治理、可扩展、可长期运行的系统。企业内部 Agent 要想真正落地，也必须从‘一次调用’升级到‘可控运行时’。”

## 七、最简版汇报摘要

可以直接复用下面这段：

“我们研究 Claude Code 后的结论是：企业内部 Agent 的核心，不是再包一层模型调用，而是建立一个可治理的 Agent 运行时。这个运行时需要具备分层启动、统一消息模型、显式 query loop、权限前置和统一扩展治理能力。Claude Code 提供了非常强的工程参考，但我们不会照搬其终端形态，而会借鉴其运行时与治理思路，结合企业内部的身份、权限、审计与集成场景来落地。”
