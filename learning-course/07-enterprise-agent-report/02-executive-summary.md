# 管理层 1 页摘要版

## 标题

`企业内部 Agent 平台设计：从 Claude Code 提炼的 5 个可借鉴原则`

## 一句话结论

如果企业内部 Agent 要真正可落地、可扩展、可治理，就不能只做“模型调用 + 几个工具”的轻封装，而应设计为一个可控运行时。

## 为什么这件事重要

企业内部 Agent 的核心问题通常不是模型能力，而是：

- 运行过程不可控
- 权限边界不清晰
- 工具和扩展越做越乱
- 长会话和任务状态难维护
- 多入口接入难统一

## Claude Code 给我们的 5 个关键启发

### 1. 分层启动与多模式入口

不同入口应分流，不应共享同样的重初始化成本。

### 2. 显式运行时主循环

Agent 不应只建模为一次请求，而应建模为一个带工具编排和恢复能力的运行时流程。

### 3. 权限前置

权限不应只是执行时审批，而应参与能力暴露和自动模式治理。

### 4. 统一消息与状态模型

统一的消息模型和状态中心，是 Agent 可审计、可恢复、可管理的基础。

### 5. 统一扩展治理

一旦进入平台化阶段，扩展能力必须统一注册和统一治理。

## 我们不会直接照搬的部分

- 不直接照搬终端 REPL 形态
- 不第一阶段就做全量平台化
- 不忽略企业的身份、审计和数据合规要求

## 建议的建设路径

### 第一期

先做：

- 单 Agent
- 核心工具
- 权限边界
- 审计留痕

### 第二期

再补：

- 长会话
- 中断恢复
- 状态管理

### 第三期

最后做：

- 插件/MCP
- 多入口统一
- 团队级技能和规范

## 最后一句话

Claude Code 最值得借鉴的，不是某个具体功能，而是它把 Agent 设计成了一个可治理、可扩展、可长期运行的系统。这正是企业内部 Agent 平台最终必须具备的形态。
