# 技术评审版

这份文档适合拿去做技术评审、架构讨论或方案评估。

它会更强调：

- 工程原则
- 模块边界
- 风险与取舍
- 落地建议

## 一、评审问题定义

如果企业要建设自己的 Agent 系统，评审时不应该只问：

- 模型选哪个
- 支持哪些工具

还应该重点问：

1. 运行时是否可控？
2. 权限边界是否前置？
3. 状态模型是否统一？
4. 扩展机制是否可治理？
5. 是否支持中断、恢复、长期会话和多入口？

## 二、从 Claude Code 中可提炼的技术原则

## 原则 1：启动分层

### 技术意义

- 降低启动成本
- 支持多模式分流
- 让轻路径不受重初始化污染

### 企业落地建议

- 将 Web、CLI、IDE、自动化入口统一抽象
- 对轻路径保留 fast-path
- 对完整运行时保留独立初始化链

### 技术风险

- 入口分支增多后测试复杂度上升

## 原则 2：显式 query loop

### 技术意义

适合统一处理：

- 流式输出
- 工具调用
- 工具结果回填
- 预算控制
- 中断与恢复

### 企业落地建议

- 不要把工具编排散在 controller/service/helper 里
- 使用统一运行时主循环或状态机承载核心执行逻辑

### 技术风险

- 运行时主循环很容易成为复杂度中心
- 需要强测试和可观测性

## 原则 3：权限前置

### 技术意义

模型看到什么能力，本身就是一种治理问题。

### 企业落地建议

- 根据身份、环境、项目、数据分级先计算能力边界
- 再构建会话级工具池
- 自动模式必须受限于明确规则

### 技术风险

- 规则系统会快速复杂化
- 需要和 IAM、审计系统打通

## 原则 4：统一消息模型

### 技术意义

消息模型不仅服务 UI，还服务：

- 工具闭环
- 恢复
- 回放
- 审计
- 多入口兼容

### 企业落地建议

- 不要只保存聊天文本
- 使用统一事件/消息模型表达用户、模型、工具和系统事件

### 技术风险

- 消息结构会逐步膨胀
- 需要边界标准化层，把内部历史和 API 输入区分开

## 原则 5：统一状态中心

### 技术意义

长期会话、多任务、多入口、插件和远程控制场景，最终都会需要统一状态模型。

### 企业落地建议

- 最少统一这些状态：
  - 会话状态
  - 权限状态
  - 任务状态
  - 扩展状态
  - 审计与通知状态

### 技术风险

- 状态中心会变大
- 需要清晰状态域划分和消费边界

## 原则 6：统一扩展注册中心

### 技术意义

多来源扩展如果没有统一收口，最终会造成平台治理失控。

### 企业落地建议

- 建立统一扩展抽象
- 所有扩展来源最终统一注册、排序、过滤、治理
- 同时区分：
  - 用户可见能力
  - 模型可调用能力
  - 外部协议资源/工具

### 技术风险

- 可见性排查链变长
- 元数据治理成本上升

## 三、企业架构建议

建议采用 4 层结构：

## 1. 入口层

职责：

- Web
- IDE
- CLI
- IM
- 自动化任务入口

## 2. 运行时层

职责：

- query loop
- 工具编排
- 中断与恢复
- 上下文预算与压缩

## 3. 治理层

职责：

- 身份与权限
- 审计
- 数据边界
- 安全策略

## 4. 扩展层

职责：

- 内建能力
- 插件
- MCP/外部协议
- 组织级技能

## 四、推荐评审问题清单

做技术评审时，建议至少问下面这些问题：

### 运行时

- 我们有没有显式主循环？
- 中断与恢复由谁负责？
- 工具结果如何进入下一轮推理？

### 权限

- 模型看到的工具集合是怎么确定的？
- 自动模式的危险能力如何约束？
- 是否有会话级权限上下文？

### 消息与状态

- 我们的历史是聊天文本，还是统一消息模型？
- 是否支持恢复与回放？
- 会话状态是否统一？

### 扩展

- 扩展来源有哪些？
- 是否统一收口？
- 用户视图、模型视图和协议视图是否区分？

## 五、不建议的方案

以下做法风险较大：

- 只做 controller 层工具调用拼接，没有 runtime loop
- 把权限放到执行末端才考虑
- 把状态散在各模块和各入口里
- 每个插件/扩展各有一套注册方式
- 过早追求全量平台化但没有治理骨架

## 六、技术评审结论模板

你可以直接复用下面这段话作为评审总结：

“我们建议企业内部 Agent 平台借鉴 Claude Code 的运行时与治理思想，而不是照搬其产品形态。重点应放在分层启动、显式 query loop、权限前置、统一消息模型、统一状态中心和统一扩展注册中心上。短期目标不是复刻一个完整平台，而是先建立一套可控、可审计、可扩展的运行时骨架，再分阶段扩展能力。”
