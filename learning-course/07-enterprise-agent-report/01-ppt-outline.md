# PPT 提纲版

这份文档适合直接转成企业内部汇报 PPT。

建议控制在 8 页以内，如果是高层汇报可以压缩到 6 页。

## 第 1 页：问题定义

标题建议：

`企业内部 Agent 平台建设，核心难点不在模型，而在运行时与治理`

要点：

- 企业 Agent 不只是“模型接个工具”
- 真正难点在：
  - 运行时可控
  - 权限治理
  - 工具与扩展治理
  - 长会话与任务状态
  - 多入口接入

建议讲法：

“如果我们只把 Agent 看成一次推理调用，短期能跑起来，但很难走向企业级稳定落地。”

## 第 2 页：为什么参考 Claude Code

标题建议：

`Claude Code 提供的不是功能清单，而是 Agent 运行时工程参考`

要点：

- 它不是简单聊天产品
- 它在工程上系统解决了：
  - 分层启动
  - query loop
  - 权限边界
  - 状态中心
  - 平台化扩展

建议讲法：

“我们不是要照搬 Claude Code 的产品形态，而是借鉴它对 Agent 工程问题的处理方式。”

## 第 3 页：最值得借鉴的 5 个设计

标题建议：

`最值得借鉴的 5 个设计原则`

要点：

1. 分层启动与多模式入口
2. 以 query loop 为核心的运行时
3. 权限前置到能力暴露层
4. 统一消息模型
5. 统一扩展注册中心

建议讲法：

“这些不是单点功能，而是让 Agent 可持续扩展、可治理、可恢复的基础设施。”

## 第 4 页：我们不会直接照搬的部分

标题建议：

`值得借鉴，但不应直接照搬的部分`

要点：

- 不直接照搬终端 REPL 形态
- 不在第一阶段做全量平台化
- 不忽略企业审计、身份和数据边界

建议讲法：

“我们借鉴的是它的运行时和治理思想，而不是逐项复制产品外形。”

## 第 5 页：目标架构分层

标题建议：

`建议的企业 Agent 平台 4 层架构`

分层：

1. 入口层
2. 运行时层
3. 治理层
4. 扩展层

每层一句解释：

- 入口层：Web/IDE/CLI/IM/自动化等多入口接入
- 运行时层：query loop、工具编排、恢复和预算控制
- 治理层：权限、审计、安全、数据边界
- 扩展层：内建能力、插件、MCP、组织技能

## 第 6 页：分期实施路线

标题建议：

`建议的三期建设路径`

阶段：

### 第一期

- 单 Agent
- 核心工具
- 权限边界
- 审计留痕

### 第二期

- 长会话
- 中断与恢复
- 任务与状态管理

### 第三期

- 插件/MCP
- 多入口统一
- 团队级技能与规范

## 第 7 页：投入与收益

标题建议：

`为什么这类建设值得投入`

收益：

- 提升 Agent 系统稳定性
- 降低后续集成混乱
- 提前建立可治理能力
- 为多场景扩展做准备

代价：

- 初期工程复杂度上升
- 需要明确边界和分期建设

建议讲法：

“这是用前期工程纪律换后期平台失控风险。”

## 第 8 页：结论

标题建议：

`结论：企业 Agent 要从“一次调用”升级到“可控运行时”`

一句总结：

“Claude Code 最值得借鉴的，不是它会不会写代码，而是它把 Agent 做成了一个可治理、可扩展、可长期运行的系统。企业内部 Agent 要真正落地，也必须从‘一次调用’升级到‘可控运行时’。”
