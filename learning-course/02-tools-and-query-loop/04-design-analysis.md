# 设计分析：为什么这样设计

## 一、为什么工具不能只是 `run(input)`

### 问题

为什么这里的工具上下文要接触 AppState、消息、通知、权限、MCP、agent？

### 回答

因为真实 agent 工具调用不是孤立计算，而是会影响：

- 会话历史
- 用户可见状态
- 当前权限决策
- 下一轮工具池
- 子代理行为

Claude Code 没有回避这种复杂性，而是把它正面编码进 `ToolUseContext`。

## 二、为什么工具池要分“真源”和“当前暴露”

### 问题

为什么需要同时存在 `getAllBaseTools()` 和 `getTools(...)`？

### 回答

因为一个成熟系统必须区分：

- 理论上的全量能力
- 当前会话允许暴露的能力

如果混成一层，就很难同时满足：

- 平台扩展性
- 运行时安全性
- 模式切换可控性

## 三、为什么 query loop 必须集中这么多逻辑

### 问题

为什么不把 compact、预算、工具执行、恢复分散到不同服务里？

### 回答

因为它们都在回答同一个问题：

“下一轮 query 该在什么消息上下文里继续？”

只要它们都影响这个问题，它们就天然属于同一条运行主线。

## 四、为什么 `State` 模式比一堆局部变量更重要

### 问题

为什么要专门建一个 `State`，而不是在循环里维护很多散变量？

### 回答

因为 continue 路径一多，散变量会让状态迁移语义变得不可读。

`State` 的价值不只是方便编码，而是把：

- 当前历史
- 当前上下文
- 当前恢复状态
- 当前继续原因

统一封装为运行时状态机。

## 五、这个设计最强的地方

### 1. 把复杂运行时行为统一在一条主线里

这让系统有能力支持：

- 流式输出
- 工具调用
- 预算控制
- 恢复与继续

### 2. 工具、消息和权限天然形成闭环

这比“模型 -> 工具 -> 日志”模式成熟得多。

### 3. 更适合长期 agent 会话

如果系统目标只是单轮 API，这套设计太重；但对 Claude Code 这种产品来说，这是合理重量。

## 六、这个设计的维护风险

### 风险 1：`ToolUseContext` 继续膨胀

当新能力不断接入时，它容易成为“万能上下文”。

### 风险 2：`query.ts` 成为理解瓶颈

继续分支和恢复逻辑越多，越需要强测试和强约束。

### 风险 3：新增工具容易跨多个边界

一个新工具可能同时涉及：

- 权限
- 消息
- UI
- 状态
- query 后续行为

接入门槛会高于普通函数型工具系统。

## 七、可迁移经验

### 什么时候值得模仿这套设计

适合：

- 多轮 agent 会话
- 工具执行复杂
- 需要恢复与继续
- 有 UI / headless 双路径

### 什么时候不值得

不适合：

- 单轮调用
- 少量工具
- 不关心恢复
- 没有长期会话

## 八、这节课真正想让你学会什么

不是记住某几个函数名，而是学会这个判断：

当你的 AI 系统开始进入“长会话 + 多工具 + 流式 + 恢复”阶段，就应该考虑 query loop 级别的运行时设计，而不是继续堆外围 helper。
